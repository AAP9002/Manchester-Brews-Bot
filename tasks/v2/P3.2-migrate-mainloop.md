## Goal
Cut over `code.py`'s if-chain dispatch to use `Navigator` and `TouchTracker`, while still pointing at v1's `MenuScreen` and `BroadcastScreen`. This proves the Navigator works without the risk of swapping in new screens at the same time.

## Context
After P1.7 the device boots with v1 screens but utilities are now properly DI'd. After P3.1 a `Navigator` class exists. This task plumbs them together.

v1 screens (`MenuScreen`, `BroadcastScreen`) currently expose:
- `get_screen()` returning the displayio Group (rename / wrap as `get_group`)
- `is_button_pressed(touch)` and `fire_button_callback(touch)` as separate calls

For this task, **wrap** rather than rewrite v1 screens. Create a tiny adapter so they fit the Navigator protocol:
- `get_group()` → return existing `get_screen()`
- `handle_touch(touch)` → if `is_button_pressed(touch)`, call `fire_button_callback(touch)`
- `on_enter(params)` → optional rebuild for the screens that need it
- `tick(now)` → no-op for v1 screens (they have nothing time-based)

Or: add the methods directly to `MenuScreen` / `BroadcastScreen` since those files are deleted in P5.1 anyway. Pick whichever is smaller.

## Files to touch

- `screens/BroadcastScreen.py` (small additions)

## Specification

In `code.py`, the main loop becomes (sketch):

```python
from utils.Navigator import Navigator
from utils.touch import TouchTracker

# ... existing wifi/display/ctp setup ...

navigator = Navigator(display)
navigator.register("menu", menu_screen)
navigator.register("broadcast", broadcast_screen)
navigator.navigate("menu")

tracker = TouchTracker(ctp)

while True:
    touch = tracker.poll()
    if touch is not None:
        navigator.handle_touch(touch)

    now = system_time.monotonic()
    navigator.tick(now)

    # 5 s wifi cadence (existing logic)
    if now - last_wifi_check > TIMING["wifi_poll"]:
        last_wifi_check = now
        # update app_state["wifi_connected"], reconnect attempt, etc.
```

Where v1 screens currently call `app_state["current_screen"] = "broadcast_screen"` to navigate, replace with `navigator.navigate("broadcast")`. Delete the if-chain in `code.py` that swaps `display.root_group`.

Keep the 5-second WiFi check, weather refresh cadence, and reconnect attempts in `code.py` for now. They survive untouched until Phase 4 when `NoConnectionScreen` takes over the offline branch.

## Notes

### Touch normalization conflict — must resolve before integration

Discovered during P2.1 smoke testing. The components are individually correct against their own specs but incompatible by convention:

- `utils.touch.TouchTracker.poll()` (P1.3) returns an **already-normalized** `(tx, ty)` tuple — it calls `normalize()` once at the source.
- `components.TouchButton.isPressed()` (v1) and `components.CardButton.is_pressed()` (P2.1) both call `utils.touch.normalize()` **internally**, expecting **raw** input from `ctp.touches[0]`.

The sketch above wires `tracker.poll()` straight through `navigator.handle_touch(touch)` into a screen's `is_button_pressed(touch)`. With either v1 `TouchButton` or v2 `CardButton`, that double-rotates and the control becomes untappable.

Pick one before doing the wiring:

1. **Drop internal `normalize()` from `TouchButton.isPressed` and `CardButton.is_pressed`.** Rely on `TouchTracker` to normalize once at the source. Cleaner: `TouchTracker` is the documented v2 main-loop input and "raw" leaks driver detail. Touches the two component files plus a one-line update to each docstring.
2. **Have `TouchTracker.poll()` return raw tuples** and let each consumer normalize. Matches v1 convention but makes `TouchTracker` little more than a rising-edge debouncer.

P2.1 was smoke-tested in isolation by passing raw `ctp.touches[0]` directly to `card.is_pressed`, deferring this decision. P3.2 cannot defer it.
